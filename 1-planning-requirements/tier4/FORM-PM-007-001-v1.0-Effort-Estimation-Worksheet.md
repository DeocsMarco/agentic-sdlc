## Purpose

This worksheet guides project managers through estimating total project effort by phase (Design, Development, Testing, Deployment) with built-in contingency calculations based on project risk level. It ensures systematic effort estimation using standard formulas and provides transparent calculations for stakeholder review.

**Used By**: Project Manager (with input from Technical Lead and team)
**Time Required**: 60-90 minutes

---

## Methodology Overview

Effort estimation uses a bottom-up approach: estimate effort for each project phase, sum the totals, and apply contingency percentage based on assessed risk level. The formula ensures realistic estimates with built-in buffer for uncertainties.

**Key Principles**:
1. **Bottom-up estimation**: Break project into phases; estimate each independently
2. **Team involvement**: Developers/testers estimate their own work (not PM alone)
3. **Historical calibration**: Use past project actuals to calibrate estimates
4. **Risk-based contingency**: Higher risk projects need higher contingency (10-30%)

**Steps**:
1. Break project into standard phases (Design, Dev, Test, Deploy)
2. For each phase, estimate effort in person-hours
3. Sum base effort across all phases
4. Assess project risk level (Low/Medium/High)
5. Apply contingency multiplier to calculate total effort

## Reference Tables

### Table 1: Standard Project Phases

| Phase | Activities Included | Typical % of Total |
|-------|---------------------|-------------------|
| **Design** | Requirements analysis, architecture, UI/UX design, technical design docs | 15-20% |
| **Development** | Coding, unit testing, code reviews, integration | 50-60% |
| **Testing** | Test case creation, QA testing, bug fixing, UAT support | 20-25% |
| **Deployment** | Release prep, deployment, documentation, training, post-launch support | 5-10% |

**Usage**: Validate your phase estimates against these typical percentages. If Development is <40% or >70%, review for estimation errors.

---

### Table 2: Risk-Based Contingency Factors

| Risk Level | Contingency % | When to Use | Example |
|------------|---------------|-------------|---------|
| **Low** | 10% | Proven technology, experienced team, clear requirements, similar to past projects | Maintenance project using existing stack |
| **Medium** | 20% | Some unknowns, moderate complexity, some new technology or team members | New feature in existing product with new API integration |
| **High** | 30% | Significant unknowns, unproven technology, inexperienced team, vague requirements | Greenfield project with experimental framework |

**Usage**: Select risk level based on WORK-PM-002 (Risk Scoring). If average risk score >5, use High; if 3-5 use Medium; if <3 use Low.

---

### Table 3: Effort Estimation Quick Reference (Person-Hours)

| Task Complexity | Small | Medium | Large |
|----------------|-------|--------|-------|
| **Simple CRUD screen** | 8-16h | 16-32h | 32-48h |
| **API integration** | 16-24h | 24-40h | 40-80h |
| **Complex algorithm** | 40-80h | 80-120h | 120-200h |
| **Database design** | 16-32h | 32-64h | 64-120h |
| **Test suite creation** | 20-30% of dev | 25-35% of dev | 30-40% of dev |

**Usage**: Use as sanity check for individual task estimates. If estimates fall far outside these ranges, review rationale.

## Common Estimation Errors & Solutions

| Error | Cause | Solution |
|-------|-------|----------|
| Estimates too optimistic (no delays expected) | Planning fallacy; ignoring risks | Always apply minimum 10% contingency; use historical actuals for calibration |
| Testing underestimated (<20% of development) | Wishful thinking; "we'll test as we go" | Enforce minimum 25% of dev effort for testing; separate QA from dev estimates |
| Missing activities (no deployment, no documentation) | Incomplete task breakdown | Use phase checklist (Table 1) to ensure all activities included |
| Wide disagreement between estimators | Different assumptions or understanding | Clarify assumptions first; use planning poker or Delphi method for consensus |
| Estimates in person-months not adjusted for team size | Assuming 1 person = 1 month of calendar time | Convert to person-hours first; use WORK-PM-005 to calculate calendar duration |
