## Purpose

This worksheet facilitates systematic risk identification across all project risk categories. It guides project teams through structured brainstorming to identify technical, resource, schedule, budget, and client-related risks, ensuring comprehensive risk coverage and reducing the chance of overlooking critical risks.

**Used By**: Project Manager (facilitating risk identification session with team)
**Time Required**: 60-90 minutes (team workshop)

---

## Methodology Overview

Risk identification uses a category-based brainstorming approach to systematically uncover potential project risks. The methodology ensures all risk dimensions are considered by working through 5 standard categories.

**Key Principles**:
1. **No risk is too small**: Capture all identified risks; prioritization comes later
2. **Focus on "what could go wrong"**: Think about threats, not solutions (mitigation comes next)
3. **Use past project lessons**: Reference historical risks from similar projects
4. **Involve diverse perspectives**: Include team members from different disciplines

**Steps**:
1. Assemble risk identification team (PM, Tech Lead, BA, QA, key stakeholders)
2. For each category, brainstorm potential risks using guiding questions
3. Document each risk with clear description and potential impact
4. Review identified risks for completeness and clarity
5. Hand off to Risk Scoring (WORK-PM-002) for prioritization

## Reference Tables

### Table 1: Risk Categories & Definitions

| Category | Definition | Example Risks |
|----------|------------|---------------|
| **Technical** | Risks related to technology, tools, architecture, integrations | Unproven technology, integration failures, performance issues |
| **Resource** | Risks related to team availability, skills, turnover | Key person unavailable, skill gaps, team turnover |
| **Schedule** | Risks related to timeline, deadlines, dependencies | Delayed deliverables, dependency bottlenecks, unrealistic estimates |
| **Budget** | Risks related to cost, funding, financial constraints | Cost overruns, scope creep, funding cuts |
| **Client/External** | Risks from client side or external dependencies | Delayed client approvals, changing requirements, vendor delays |

**Usage**: Work through each category systematically; aim for minimum 2-3 risks per category.

---

### Table 2: Risk Description Quality Checklist

| Quality Criterion | Good Example | Poor Example |
|-------------------|--------------|--------------|
| **Specific** | "Database migration fails due to schema incompatibility" | "Database issues" |
| **Clear Cause** | "Key developer leaves mid-project due to competing offer" | "Resource problem" |
| **Impact Stated** | "Client delays approval for 3 weeks, pushing delivery date" | "Client might delay" |

**Usage**: After writing each risk, validate against this checklist to ensure clarity.

## Common Risk Identification Errors & Solutions

| Error | Cause | Solution |
|-------|-------|----------|
| Risks are too vague ("technical issues") | Lack of specificity | Ask "What specific technical issue? What would cause it?" |
| Solutions instead of risks ("need backup server") | Jumping to mitigation too early | Rephrase as risk: "Primary server failure causes 24-hour outage" |
| Only high-level risks identified | Not digging deep enough | For each risk, ask "What else could go wrong in this area?" |
| One category dominates (e.g., all Technical) | Facilitation bias or team composition | Ensure balanced team; spend equal time on each category |
| Risks are actually issues (already happening) | Confusing current problems with future risks | Separate current issues into issue log; focus risks on future uncertainties |
